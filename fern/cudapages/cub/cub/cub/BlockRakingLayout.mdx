BlockRakingLayout provides a conflict-free shared memory layout abstraction for 1D raking across thread block data.

<Accordion title="Template parameters">

</ParamField>

| `SHARED_ELEMENTS` <Badge intent="note" minimal>static</Badge> <Badge intent="note" minimal>constexpr</Badge> | `int` | The total number of elements that need to be cooperatively reduced. |

| `SEGMENT_LENGTH` <Badge intent="note" minimal>static</Badge> <Badge intent="note" minimal>constexpr</Badge> | `int` | Number of raking elements per warp-synchronous raking thread (rounded up). |
| `RAKING_THREADS` <Badge intent="note" minimal>static</Badge> <Badge intent="note" minimal>constexpr</Badge> | `int` | Never use a raking thread that will have no valid data (e.g., when BlockThreads is 62 and SEGMENT_LENGTH is 2, we should only use 31 raking threads). |
| `HAS_CONFLICTS` <Badge intent="note" minimal>static</Badge> <Badge intent="note" minimal>constexpr</Badge> | `bool` | Whether we will have bank conflicts (technically we should find out if the GCD is &gt; 1). |

| `USE_SEGMENT_PADDING` <Badge intent="note" minimal>static</Badge> <Badge intent="note" minimal>constexpr</Badge> | `bool` | Pad each segment length with one element if segment length is not relatively prime to warp size and can't be optimized as a vector load. |
